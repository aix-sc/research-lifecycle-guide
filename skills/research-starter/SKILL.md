# research-starter — 研究スタート伴走（start / update 両モード）

## 概念

研究スタートガイド v9 の6フェーズを**スロット行列（状態機械）**として使う研究開始コンパイラ。
`ヒアリング（会話・メモ）→ スロットへの写像 → 埋まらないスロットを❌検出 → 充足質問に回答 → DOCX更新` のループを回し、**すべての❌/⚠️が✅になれば研究開始の環境整備が完了する**ことを保証する。

このため、**❌/⚠️を出すときは必ず「充足質問」または「具体的な実行手順（URLつき）」を添える**。「何が欠けているか」だけの指摘は禁止。ユーザーの手を止めないことが本スキルの目的である。

## 入力とモード

| モード | 条件 | 動作 |
|---|---|---|
| **start** | DOCXが未添付（会話・メモのみ） | `assets/research_start_template_v0_JA.docx` を雛形として新規生成 |
| **update** | `{YYYYMMDD}_ResearchStart_@{NAME}_{LANG}_vX.docx` が添付 | 進捗を検査し、更新履歴に追記して vX を +1 |

v0テンプレートが無くても必ず起動する。会話から読み取れる情報はすべて先にスロットへ写像し、**確定できない項目だけを質問する**。

## スロット行列（判定は ✅ / ❌ / ⚠️ / N/A）

**順番が戦略**：Phase 01→06 の順に検査し、次にやるべき Phase を常に1つだけ提示する（先のPhaseの質問を先行して浴びせない）。

| # | スロット | 合格条件 | 充足質問の例 |

## 実行手順

### start モード
0. **Phase 04 の分業**：虎の巻（P4-1〜P4-4）に着手する際は `literature-researcher` スキルを呼び出して探索（トップ研究者・トピック地図・世界線・候補文献）を行い、候補リストを `literature-reviewer` に引き継いで検証・要約・URL確認を行う。結果は DOCX の Phase 04 欄に記入する。
1. **会話・メモをスロットへ写像**し、写像表（どの情報がどのスロットに入ったか）を先に示す。
2. **ヒアリング**：埋まらないスロットのうち、**現在のPhaseに属するものだけ**をまとめて1回で質問する。答えられない項目は〔　〕のまま進めてよい（でっち上げ禁止）。
3. **DOCX生成**：雛形を複製し、埋まったスロットを記入。命名は `{YYYYMMDD}_ResearchStart_@{NAME}_{LANG}_v0.docx`。
4. **ギャップレポート**：✅（できている点を必ず言語化して褒める）／❌／⚠️を提示。各❌/⚠️に充足質問または実行手順を添える。優先度は**Phase順**（Phase 01 の❌が最優先）。
5. **次の1アクション**を具体的に1つだけ提示する（例：「今日は orcid.org での登録だけ完了させましょう」）。

### update モード
1. 添付DOCXの全スロットを再検査し、前版から✅に転じた項目を冒頭でハイライトする。
2. **末尾の更新履歴テーブルの最上段**（表内の一番上の行＝最新版が上）に当該版の行を3項目の箇条書きで追記する：①✅完了したPhase・チェック項目 ②🔄更新した問い・仮説（Q5〜Q7相当） ③▶次のアクション。既存の行は編集しない。
3. vX を +1、日付を当日に更新して出力する。

## アドバイス用の鉄則（スロットを埋める際の判断基準）

- **ORCID**：研究者ライセンス。主要出版社が著者管理に使用。「これがないと存在しないに等しい」。
- **問い・仮説**：いつでも更新してOK。定期的な書き直しは「考え方の筋トレ」。
- **磁石戦略**：LP・名刺・MVPで「XをしているYさん」ブランドを作ると、情報・紹介・共同研究・ユーザーが引き寄せられる。事前登録者数は研究根拠になる定量データ。
- **7割・2020年以降の法則**：査読者はリファレンスのビジュアルで質を判断。最新7割＋巨匠数本＋データ原典。探索は literature-researcher、検証は literature-reviewer が担当する。
- **arXiv**：原則引用不可（査読未了）だが **AI/ML分野は preprint と明記すれば引用可**。arXivへの投稿自体は速報性のための業界標準。
- **投稿戦略**：投稿先誌の最新号をイントロ早期に引用／ショート・コミュニケーション（短報）は最初の論文に最適／入り切らない文献は Supplementary へ。

### 更新箇所には必ずコメントを付す（改稿のたびに必須）

**vX → vX+1 で本文を変更（新規生成・加筆・書き換え・削除）したすべての箇所に、Wordコメントで「何を・なぜ変えたか」を残す。** ユーザーが差分を追えないまま新版を受け取ることがないようにするための必須規定であり、省略は違反。

- **付与単位**：変更した段落・項目・表セルごとに1コメント（連続する小さな変更はまとめてよいが、章・セクションをまたいでまとめない）。
- **コメント本文の書式**（1〜3行、平易な言葉）：

  ```
  [vX 更新] 何を変えたか（1行）
  理由：なぜ変えたか（スロットID・充足質問への回答など根拠を明示）
  ※ 必要なら：他セクションへの波及
  ```

- 語句・文の**書き換え／削除**は変更履歴（tracked changes）で記録し、**その箇所にコメントで理由を添える**。**新規生成**した項目は冒頭にコメントで `[vX 更新] 新規生成` と根拠を示す。**充足された❌/⚠️** には `❌→✅` と、どの充足質問の回答を反映したかを書く。
- 前版のコメントは削除しない（履歴として残す）。
- 著者名は必ず `Yusuke Takahashi, PhD`（`comment.py --author "Yusuke Takahashi, PhD" --initials "YT"`、変更履歴は `w:author="Yusuke Takahashi, PhD"`）とする（"Claude" やスキル名は使わない）。
- 出力前に `validate.py --original` で検証する（tracked changes を含むため必須）。

## DOCXの運用

- 3部構成を維持する：**①初期質問シート → 改ページ → ②フェーズ別ドリル → 改ページ → ③更新履歴（文書の最後尾）**。
- **更新履歴は必ず文書の最後尾に置き、その直前で改ページする**（ドリル本文の可読性を優先し、履歴は巻末資料として扱う）。既存文書で履歴が前方にある場合は、更新時に末尾へ移動する。
- 新規生成は docx-js、既存編集は docx スキルの手順（unpack → XML編集 → pack）に従い、出力前に `validate.py` で検証する。
- ファイル命名（file-naming-convention 準拠・本スキル専用の拡張形）：

  ```
  {YYYYMMDD}_ResearchStart_@{NAME}_{LANG}_vX.docx
  例：20260709_ResearchStart_@Tanaka_JA_v0.docx
  ```

- JA / EN 両方を運用する場合は同一バージョンで揃える。

## 収束の定義

Phase 01〜06 のすべてが✅になったら「研究開始の環境整備 完了」を宣言し、**paper_template_v0 の1ページ目の記入 ＋ paper-compiler の起動**（論文執筆フェーズへの移行）を提案する。

## 禁止事項

1. でっち上げ禁止。ユーザーが答えていない項目を勝手に埋めない（〔　〕で残す）。
2. 指摘だけの❌/⚠️は禁止。必ず充足質問か実行手順を添える。
2b. 改稿で本文を変更したのに、変更箇所へWordコメント（[vX 更新] 何を・なぜ）を付けないのは違反。
3. Phaseを飛ばす提案をしない（順番が戦略）。
4. 一度に全Phaseの質問を浴びせない。現在のPhaseに集中する。
