# research-proposer — 研究プロポーザル・コンパイラ（write / review 両モード）

## 概念

研究プロジェクト計画書の固定目次を**スロット行列（状態機械）**として使うプロポーザル・コンパイラ。
`素材（会話・メモ・議事録・文字起こし）→ スロットへの写像 → 埋まらないスロットを❌検出 → 充足質問に回答 → 再コンパイル` のループを回し、**すべての❌/⚠️を潰せばキックオフ可能な計画書が完成する（チェックリスト＝十分条件）**ことを保証する。

**❌/⚠️を出すときは必ず「充足質問」または「既存素材からの下書き案」を添える**。「何が欠けているか」だけの指摘は禁止。

## 入力とモード

| モード | 条件 | 動作 |
|---|---|---|
| **write** | DOCXが未添付（会話・メモ・議事録が主入力） | `assets/proposal_template_v0_JA.docx` を雛形として新規生成＋ギャップ検出 |
| **review** | 既存の Proposal vX.docx が添付 | スロット検査＋充足質問。回答を受けたら write に戻って再コンパイル |

v0雛形が無くても必ず起動する。素材から読み取れる情報はすべて先にスロットへ写像し、**確定できない項目だけを質問する**。空きスロットが多いときだけ、1回まとめて確認質問する。

## 固定目次（この順序を厳守）

冒頭ブロックは「誰が・何を・いつ」を最優先で示すため、**概要の直後にスケジュール**を置く。

```
タイトル（プロジェクト名・版数・日付・著者ライン）
─ 研究者リストと所属
─ 研究の概要（要旨）
─ スケジュールと役割     ← 概要の直後（タスク確認を最優先）
─ 研究の背景
1. 研究の目的・リサーチクエスチョン・中核貢献
2. ポートフォリオ：成果物構成と著者割り当て
3. 実験方針・研究方法
4. 発表・投稿戦略（学会／ジャーナルの選定と適合）
5. 投稿先・資料・リンク
6. FAQ・想定問答
7. Literature Review（関連研究）※literature-reviewer スキルが生成し、最後（FAQの後）に改ページで配置
```

## スロット行列（判定は ✅ / ❌ / ⚠️ / N/A）

| # | スロット | 合格条件 | 充足質問の例 |
|---|---|---|---|
| S1-1 | 研究者リスト | 氏名・所属・役割・識別子（ORCID/e-Rad）が表で | メンバー全員の氏名・所属・役割を教えてください |
| S1-2 | 著者順の根拠 | 強み適合＋学術上の役割を一言 | 筆頭は誰で、その理由は？ |
| S2-1 | 概要（要旨） | 何を・なぜ・どう行い・何が得られるかを1段落 | このプロジェクトを1段落で説明すると？ |
| S2-2 | 中核主張 | 平易な1文 | 最終的に言いたいことを一文で言うと？ |
| S2-3 | キーワード・分野 | 3〜5語＋想定分野（例：cs.DB） | キーワードと投稿カテゴリは？ |
| S3-1 | スケジュール表 | 成果物×時期×役割×発表の表 | 各成果物の締切・通知・カメラレディの日付は？ |
| S3-2 | go/no-go | 直近の判断点を先頭で明示 | 直近で「やる/やらない」を決めるポイントは？ |
| S3-3 | 依存関係 | 担当者間の依存と現在ステータス | 誰の作業が誰の前提になっていますか？ |
| S4 | 背景 | 問題と現状／既存手法の限界／動機・新規性／関連研究 | 既存の方法では何ができませんか？ |
| S5-1 | 目的・RQ | 研究目的とRQを明示 | このプロジェクトのRQを1〜3個挙げると？ |
| S5-2 | 中核貢献 | 別個の貢献を列挙 | 世界に対する新しい貢献は何ですか？ |
| S6-1 | ポートフォリオ | 成果物ごとに会場（締切）／著者順／別個の貢献 | 何本の論文に分け、それぞれどこへ出しますか？ |
| S6-2 | 非サラミ根拠 | 各貢献の独立性と相互引用関係 | 各論文が「別の貢献」である理由は？ |
| S6-3 | プレプリント方針 | 先取権・版管理・ライセンス | arXiv 投稿のタイミングと方針は？ |
| S7-1 | 実験一覧 | 実験ID・概要・対象成果物・担当の表 | どんな実験を、誰が、どの論文のために行いますか？ |
| S7-2 | 評価指標 | 指標と「得たい結果」 | 何がどうなれば成功ですか？ |
| S7-3 | 実施状況 | 完了／パイロット済み／未計画の区別 | パイロットは実施済みですか？結果は？ |
| S8-1 | 候補会場 | 日程・索引・テンプレ・投稿システム・締切 | 候補会場のCfPは確認しましたか？ |
| S8-2 | 適合理由 | 会場ごとの適合・注意点（W-blind等） | なぜその会場が合っていますか？ |
| S8-3 | 役割分担 | フラッグシップ／速報／ジャーナルの分担 | どの成果物がフラッグシップですか？ |
| S9 | リンク | CfP・ポータル・テンプレ・companion/OSS（DOI方針） | 再現用リポジトリの公開方針は？ |
| S10-1 | FAQ | 平易な言葉（中学生でも分かる水準）で回答 | 家族に聞かれたら何と説明しますか？ |
| S10-2 | 想定問答 | 用語定義＋方法論的批判への先回り回答 | 査読者が最初に突くとしたらどこですか？ |

## writeモードの実行手順

1. **素材→スロット写像表の提示**：メモ・議事録の各項目がどのスロットに入るかの対応表を先に示す（ユーザーが写像の誤りを訂正できるようにする）。
2. **スロット生成**：埋められるスロットは全文を生成する。プレースホルダは〔　〕で示す。記述要件（下記）を全スロットで遵守する。
3. **ギャップレポート**：✅（良くできている点を必ず言語化）／❌／⚠️を提示し、各行に充足質問または下書き案を添える。優先度 **HIGH（キックオフの成立に必須：S1〜S3・S5）／ MEDIUM（S4・S6〜S8）／ LOW（S9〜S10）**を付す。
4. **Literature Review の付加**：`literature-reviewer` スキルを呼び出し、関連研究のリスト（書誌／中学生でもわかる数行の概要／本文URL、URL未確認は ⚠️ 出典不明・要確認）を生成して、**「6. FAQ・想定問答」の後に改ページで**最終セクションとして配置する。改稿時は同セクションを上書き更新する。
5. **DOCX出力**：雛形を複製して記入し、下記の命名・履歴ルールに従って出力する。

## reviewモードの実行手順

1. 添付DOCXの全スロットをルーブリックとして検査し、前版から✅に転じた項目を冒頭でハイライトする。
2. 出力フォーマット：

```
## 📋 research-proposer レビュー結果
**プロジェクト名**／**版**：vX

| # | スロット | 判定 | 根拠（本文引用） | 充足質問 / 下書き案 |
|---|---|---|---|---|
（全スロット。❌/⚠️の行は充足質問欄を必ず埋める）

## 🎯 収束レポート
- ✅ ハイライト（必ず冒頭に置く）
- 残り❌：N件 / ⚠️：M件（前版比）
- 優先度 HIGH / MEDIUM / LOW 別の次アクション
```

3. 本文への個別の修正提案は、docx スキルの手順に従い **Wordコメント（理由・質問）＋変更履歴（書き換え案）**として該当箇所にアンカーする。本文中に〔コメント：…〕のような注記を書き込まない。
4. 回答・加筆を受けたら write に戻って再コンパイルし、vX を +1 する。

## 記述要件（スロットを埋める際の基準）

- **研究者リスト**：表形式。役割（PI／責任著者／実装担当 等）と識別子を明記。
- **スケジュールと役割**：成果物ごとに「項目・時期／各メンバーの役割／発表（日程・渡航・発表者）」。締切・通知・カメラレディを明記し、go/no-go 判断点を先頭で目立たせる。
- **概要**：abstract 相当の1段落＋中核主張1文＋キーワード。
- **ポートフォリオ**：速報性・到達と権威・深さと永続性などの目的に成果物を対応づける。
- **実験方針**：実験ID・タイトル・概要・対象成果物・担当を一覧化し、評価指標と「得たい結果」を明示。
- **FAQ**：専門用語の定義を含め、中学生でも分かる言葉で。方法論的批判への先回り回答を含める。
- 未記入箇所は〔　〕、記述要件のガイド行は内容確定後に削除する。

## DOCXの運用

- 新規生成は雛形（`assets/proposal_template_v0_JA.docx`）の複製から。docx の編集は docx スキルの手順（unpack → XML編集 → pack）に従い、出力前に `validate.py` で検証する。TOC は Word 側で F9 更新。
- 生成するDOCXの2ページ目に**更新履歴テーブル**（`vX（日時）｜更新内容`）を設け、コンパイル/レビューのたびに最上段へ当該版の行を4項目の箇条書き（✅良い点／❌⚠️件数（前版比）／HIGH指摘／主要な充足質問）で追記する。**キックオフ配布版では更新履歴ページを削除**する。
- ファイル命名（file-naming-convention 準拠・本スキル専用の拡張形）：

  ```
  {YYYYMMDD}_{PROJECT_SHORT_NAME}_Proposal_@{PI}_{LANG}_vX.docx
  例：20260709_TimeAxisSynth_Proposal_@Takahashi_JA_v0.docx
  ```

- 改稿のたびに vX を +1 し、日付を当日に更新する。JA / EN 両方を運用する場合は同一バージョンで揃える。

## 収束の定義

❌＝0 かつ HIGH の⚠️＝0 になったら「キックオフ可能版」到達を宣言し、①更新履歴ページの削除、②メンバーへの配布、③各成果物の paper_template_v0 起票（paper-compiler への接続）を提案する。

## 禁止事項

1. でっち上げ禁止。実験結果・数値・締切・共著者名はユーザーが与えない限り生成しない（〔　〕で残す）。
2. 指摘だけの❌/⚠️は禁止。必ず充足質問か下書き案を添える。
3. 固定目次の順序を変えない。箱にない章を足さない。
4. 空きスロットが多いときも、確認質問は1回にまとめる。
