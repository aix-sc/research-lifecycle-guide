# paper-compiler v2 — 論文コンパイラ（write / review / audit 3モード）

## 概念

チェックリストを**状態機械**として使う論文コンパイラ。
`素材（箇条書き・メモ・文字起こし）→ スロット行列への写像 → 埋まらないスロットを❌検出 → ユーザーが充足質問に回答 → 再コンパイル` のループを回し、**すべての❌/⚠️を潰せば論文が完成する（チェックリスト＝十分条件）**ことを保証する。

このため、**❌/⚠️を出すときは必ず「充足質問」または「既存素材からの下書き案」を添える**。「何が欠けているか」だけの指摘は禁止。ユーザーに執筆労力を割かせないことが本スキルの目的である。

## モード判定（最初に必ず行う）

| 入力・トリガーの例 | モード | 参照 |
|---|---|---|
| 素材のみ／「論文にして」「このメモから論文」 | **write**：生成＋ギャップ検出 | 骨格のスロット行列＋`evaluation-planner.md` |
| ドラフトvXあり／「足りないものを教えて」「reviewモードで」 | **review**：スロット検査→充足→再コンパイル | 同上 |
| 完成稿（PDF/原稿）／「論文を確認して」「投稿前チェック」「IMRADチェック」 | **audit**：単発の18項目監査 | `paper-auditor.md` |

- review と audit の違い：review＝**収束ループ**（改稿前提・DOCX更新を伴う）、audit＝**単発合否ゲート**（投稿直前・その場で完結）。
- audit で❌が3件以上 or HIGH指摘がある場合は、review ループへの移行を提案する（`paper-auditor.md` 参照）。

## 入力の3軸

初回に paper_template_vX.docx の1ページ目（初期質問テーブル）から読み取る。未添付の場合は会話・素材から推定し、確定できない軸のみ質問する。

### 軸1：提案形式（型）

| 型 | 核となる貢献 | Methods/提案章の内部構造 |
|---|---|---|
| **A** アーキテクチャ提案型 | 概念モデル＋構成要素＋主関数 | 概念 → N要素分解 → Primitive Functions → 統合写像 |
| **B** アルゴリズム型 | 手続き（パイプライン） | ステップ列Overview → 各要素定義 → 入出力データ構造 → 統合関数 |
| **C** システム/アーティファクト型 | 動くシステム全体の設計判断 | 要求 → 設計判断と根拠 → アーキテクチャ → 実装 → 運用知見 |
| **D** 実証研究型 | 現象の観察・測定からの知見 | RQ/仮説 → 研究デザイン → 参加者/データ → 手続き → 分析手法 → 倫理 |

**A/B判定プロトコル（流し込みエディタ・浦木 由来）**：システム／概念モデル／構成要素が中心 → A、手順／ステップ／入力→計算→出力が中心 → B。推定した型を**出力の冒頭に明示**し、判断が割れる場合のみ一度だけ確認する。選んだ型のブロックだけを使い、**箱にない構成を足さない**（詳細は `paper-composition-checker.md`）。

### 軸2：研究フェーズ（＝評価の契約）

| フェーズ | 必要な証拠 | 許される主張 | 禁止される主張 |
|---|---|---|---|
| **P1** ポジション | 予備実験・概念実証・シナリオ | feasibility・方向性の妥当性 | 有効性・優位性の断定 |
| **P2** 実装 | 動作するシステム＋定性的ユーザー評価 | 実現可能性＋初期ユーザー知見 | 統計的一般化 |
| **P3** 提案 | 定量・統計的評価（ベースライン比較・検定・アブレーション） | 有効性・優位性 | — |
| **P4** 領域応用 | 定性＋定量の混合評価・ドメイン専門家の関与 | 実領域での有用性・転移可能性 | — |

### 軸3：投稿先（venue）

`国内研究会/WS` → `国際会議` → `トップ会議・ジャーナル` → `EJC・IOS Press系（清木系）` の4区分、または venue 名の直接指定。フェーズ×venue の実験・評価アドバイスは `evaluation-planner.md` を必ず読んで導出する。

## 骨格の選択ルール

1. **デフォルトは IMRAD**（`paper-blueprint-designer.md` を読む。アブスト7文法＝閉世界レシピを内蔵）。
2. **箱テンプレ5章構成**（アブスト/1背景/2提案/3実現/4実験/5まとめ）を使うのは、**型が A または B、かつ venue が EJC・IOS Press系（清木リネージ）**のとき、または明示指定があったときのみ（`paper-composition-checker.md` と `paper-composer.md` を読む）。
3. 型 C・D は常に IMRAD。骨格を混在させない。
4. 骨格が違ってもスロット行列は共通語彙（何を書くべきかの単位）であり、write/review は同一機構の裏表として動かす。

## writeモードの実行手順

1. **3軸＋モードの確定**：テンプレ1ページ目 or 会話から読み取る。骨格（と箱テンプレの場合は採用バージョンA/B）を宣言する。
2. **対応する参照ファイルを読む**：骨格のスロット行列＋ `evaluation-planner.md`。
3. **素材→スロット写像表の提示**：添付されたメモ・文字起こしの各項目が、どのスロットに入るかの対応表を先に示す（ユーザーが写像の誤りを訂正できるようにする）。
4. **スロット生成**：埋められるスロットは、骨格の形式文体（IMRADならアブスト7文法、箱テンプレなら定型句・signature動詞・"X indicates"構文まで）で**全文を生成**する。プレースホルダは〔　〕で示す。
5. **ギャップレポート**：合格したスロットは✅として明示する（良くできている点を必ず言語化する）。埋められないスロットを ❌、部分的なものを ⚠️ とし、**各行に充足質問（ユーザーが答えれば✅になる具体的な質問）または下書き案**を添える。優先度 HIGH（投稿の成立に必須）/ MEDIUM / LOW を付す。
6. **実験・評価アドバイス**：`evaluation-planner.md` の「型×フェーズ×venue」から、必要な実験セット・評価指標・サンプル規模・倫理記述の要否を具体的に提示する。
7. **DOCX出力**：下記「テンプレートDOCXの運用」に従い vX を繰り上げ、**「レビュー履歴」テーブルの最上段に当該版の行を箇条書き4項目（✅／件数／HIGH／充足質問）で追記**して出力する。本文への個別の修正提案・充足質問は「本文への修正提案」の規定どおりWordコメント・変更履歴として該当箇所に付与する。

### 要入力スロットの網掛け規定（旧・色分けHTMLの後継）

- 埋められない〔　〕スロット（❌/⚠️付き）には、DOCX内で**黄色の文字網掛け（`w:highlight="yellow"`）**を付け、充足質問をコメントとしてアンカーする。ドラフトを開いた瞬間に「あとどこを埋めれば完成か」が視覚的に分かる状態を保つ。
- スロットが充足されたら網掛けを解除する。**投稿版（submission / camera-ready）では黄色網掛けが残っていないことを確認**する（残存＝未充足スロットの検出にも使える）。
- 旧・流し込みエディタの色分けHTML出力（グレー＝箱／青＝アイデア由来）は廃止。定型部分の識別は網掛けではなく `paper-composer.md`（原本）との照合で行う。

## reviewモードの実行手順

1. 3軸を確定し、対応するスロット行列をルーブリックとしてドラフト全文に適用する。
2. 判定記号は全モード共通：

| 記号 | 意味 |
|---|---|
| ✅ | 合格（本文に明確な根拠） |
| ❌ | 欠落（スロットが空） |
| ⚠️ | 部分的・要確認 |
| N/A | 型・フェーズ・venue上対象外 |

3. 出力フォーマット：

```
## 📋 paper-compiler レビュー結果
**タイトル**／**型・フェーズ・venue・骨格**／**版**：vX

| # | スロット | 判定 | 根拠（本文引用） | 充足質問 / 下書き案 |
|---|---|---|---|---|
（骨格別スロット行列の全項目。❌/⚠️の行は充足質問欄を必ず埋める）

## 🎯 収束レポート
- ✅ ハイライト：今回よくできている点・前版から✅に転じた項目（必ず冒頭に置く）
- 残り❌：N件 / ⚠️：M件（前版比）
- 優先度 HIGH / MEDIUM / LOW 別の次アクション
- 実験・評価に関するアドバイス（フェーズ×venue由来）
```

4. **レビュー履歴への記録**：レビュー結果の要約を、DOCXの「レビュー履歴」テーブルの最上段に当該版の行として**箇条書き4項目（✅／件数／HIGH／充足質問）**で追記する（詳細は「テンプレートDOCXの運用」参照）。
5. **収束の定義**：❌=0 かつ HIGH の⚠️=0 になったら「投稿可能版」到達を宣言し、投稿版の整形（初期質問テーブル・レビュー履歴ページの削除・黄色網掛けゼロの確認・venueテンプレへの流し込み）を提案する。

## auditモードの実行手順

1. `paper-auditor.md` を必ず読み、その手順・18項目・出力フォーマットに従って単発監査を行う（項目18＝アブスト7文法・閉世界チェック）。
2. 骨格が箱テンプレ5章の場合、IMRAD専用項目（09・15）は N/A とし、`paper-composition-checker.md` の H-X 整合チェックの実施を提案する。
3. 監査は単発で完結する（DOCX更新・レビュー履歴追記は行わない）。**❌が3件以上 or HIGH指摘あり**の場合のみ、reviewモードへの移行を提案する。

## 本文への修正提案：Wordコメント＋変更履歴で行う（インライン注記の禁止）

ドラフト本文に対する個別の修正提案は、**本文中に〔コメント：…〕のような注記を書き込まず**、DOCXのネイティブ機能で該当箇所にアンカーする。docx スキルの手順に従う：

- **著者名の規定**：コメント・変更履歴の著者は**必ず `Yusuke Takahashi`** とする（"Claude" や "paper-compiler" を使わない）。変更履歴は `w:author="Yusuke Takahashi"`、コメントは `comment.py --author "Yusuke Takahashi" --initials "YT"` で付与する。
- **語句・文の書き換え提案** → **変更履歴（tracked changes）**：`<w:ins>`（挿入）／`<w:del>`（削除、テキスト要素は `<w:delText>`）で `w:author="Yusuke Takahashi"`・`w:date` を付けて記録する。ユーザーがWord上で承認/却下できる形にする。
- **理由説明・充足質問・代替案の提示** → **コメント機能**：docx スキルの `scripts/comment.py` で comments.xml 群を生成し、`<w:commentRangeStart/End>`・`<w:commentReference>` を該当テキストに配置してアンカーする。特定箇所に紐づく充足質問はギャップレポートだけでなく、その箇所へのコメントとしても付与する。
- **スロット自体が欠落している場合**のみ、本文のあるべき位置に〔　〕プレースホルダ（❌マーク＋黄色網掛け付き）を置き、そこにコメントで充足質問をアンカーする。
- レビュー履歴テーブルには要約のみを書き、個別提案の本文はコメント/変更履歴側に置く（二重管理しない）。
- 出力前に `validate.py --original` で検証する（tracked changes を含むため必須）。

## テンプレートDOCXの運用（paper_template）

- 雛形は `assets/paper_template_v0.docx`。運用時は次の**4部構成**とする：**①初期質問テーブル**（3軸・タイトル・手法名・中核主張・著者・締切・素材一覧）→ 改ページ →**②Literature Review（関連研究）**（`literature-reviewer` スキルが生成・更新。「初期質問テーブル」と「レビュー履歴」の間に改ページで挿入する）→ 改ページ →**③「レビュー履歴」テーブル**（2列：`vX（日時）｜レビュー内容`）→ 改ページ →**④論文ドラフト領域**。
- **literature-reviewer の呼び出し**：write の初回コンパイル時（および文献に変更があった改稿時）に literature-reviewer スキルを呼び出し、②を生成・上書き更新する。各文献は「書誌／中学生でもわかる数行の概要／本文URL」の3点セット。URL未確認は ⚠️ 出典不明・要確認。投稿版では ✅ の文献を本文 References へ転記して②を削除し、⚠️ が残る文献は転記しない。
- ユーザーは1ページ目を記入した paper_template と、議事録・文字起こし・メモを添付して本スキルを起動する。
- **改稿のたびに vX を+1**し、**初期質問テーブルとレビュー履歴は投稿版まで常に維持**する。ドラフトはドラフト領域を書き換える。
- **レビュー履歴の記録ルール**：コンパイル/レビューを行うたびに、**ヘッダー直下（最上段）に新しい版の行を追加**する（v2の行がv1の行の上に来る＝新しい版ほど上）。`vX（日時）`列には版番号と実施日時を記載する。`レビュー内容`列は**1行の連続文にせず、必ず次の4項目の箇条書き**で書く：
  1. ✅ 良くできている点・合格したスロット（**うまくできている点は必ず✅として明示的に褒める**。前版から改善した項目も含む）
  2. ❌／⚠️ の件数（前版比の増減付き）
  3. 優先度HIGHの指摘
  4. 主要な充足質問
  既存の行は編集せず履歴として保存する。
- **投稿版（submission / camera-ready）では初期質問テーブル・Literature Review・「レビュー履歴」の各ページを削除**する（Literature Review は転記後に削除）。
- **ファイル命名規則**（file-naming-convention 準拠・本スキル専用の拡張形）：

  ```
  {YYYYMMDD}_#{VENUE}_{SHORT_TITLE}_@{FIRST_AUTHOR}_{LANG}_v{X}.docx
  ```

  例：`20260706_#EJC2027_TimeAxisSynth_@Takahashi_JA_v0.docx`
  - `YYYYMMDD`＝最新版を出力した当日の日付（改稿ごとに更新）
  - `#{VENUE}`＝投稿先（Q4 から。例：`#EJC2027` `#iCAST2026` `#CSCW2027`。未定なら `#TBD`）
  - `SHORT_TITLE`＝短縮タイトル（英数字・アンダースコア、空白なし）
  - `@{FIRST_AUTHOR}`＝筆頭著者の姓（例：`@Takahashi`）
  - `LANG`＝`JA`/`EN`（対訳セットは日付・版を揃え、言語のみ変える）
  - `v0`＝質問シート記入済み・初回コンパイル前。以降、改稿ごとに +1
  - 雛形自体の名称 `paper_template_v0.docx`（Skill内アセット）は例外的に固定。プロジェクト開始時に複製し、上記パターンに改名して使う。
- DOCX の生成・編集は docx スキルの手順（新規=docx-js／既存編集=unpack→XML→pack→validate）に従う。

## 他スキルとの連携・振り分け

- **paper-review-imrad（旧・JA/EN）**：本スキル v2 の auditモードに**吸収済み**。旧スキルが同一環境に登録されている場合は起動が競合するため、**削除を推奨**する。
- **literature-reviewer**：参考文献の調査・平易な要約・URL検証を担う独立スキル。write モードから呼び出し、paper_template の「初期質問テーブル」と「レビュー履歴」の間に改ページで Literature Review（関連研究）を挿入・更新させる。audit で項目08〜11に⚠️が出た場合も呼び出しを提案する。
- **research-starter**：最上流（研究開始の環境整備）。全Phase完了後に本スキルへ接続される。
- **research-proposer**：上流（複数論文のポートフォリオ・投稿戦略・プロジェクト計画書）。本スキルは単一論文のコンパイルを担当。
- **docx / file-naming-convention**：出力時に必ず適用。
- **skill-creator**：本スキルの評価・洗練（学生論文＝既知欠陥の検出率、優良採択論文＝偽❌率の2指標）を行うときに使用。

## 重要な注意
