# paper-blueprint-designer：IMRAD骨格スロット行列（設計図・デフォルト）

各スロットは `ID｜何を書くか｜✅の判定基準｜充足質問の例` で定義する。writeモードではこの順に生成し、reviewモードではこの順に検査する。型（A/B/C/D）で差し替わるのは **Mブロックのみ**。フェーズ（P1〜P4）と venue で差し替わるのは **Eブロックの要求水準**（`evaluation-planner.md` 参照）。

## アブスト7文法（浦木）─ T2・I1〜I3 の生成レシピ

Abstract と Introduction 前半は、以下の**7文の論理展開**をベースに生成する。狙いは**適切な「閉世界」を作る**こと：膨大な関連研究や周辺概況を背景に取り込むと世界観が必要以上に広がり、説明の完全性を定義できなくなる。前半4文で完全な世界観を先に閉じ、その中で提案を説明する。

| 文 | 内容 | 対応スロット |
|---|---|---|
| 1 | 一般に、人間（脳のプロセス・機能・枠組み）なら こうする ※人間にできないことへの挑戦なら「人間にはこれが難しい」と定義 | T2前半 / I1 |
| 2 | しかしながら、それがまだシステム上で実現できていない | T2前半 / I2① |
| 3 | そのため、上記の実現は本分野における重要な課題である | T2前半 / I2 |
| 4 | 我々は（論文タイトル）を提案する | T2 / I3 |
| 5 | 本研究の特徴は〜（1文目の機能を実現するための特徴）〜する点にある | T2 / I3〜I4 |
| 6 | 本研究の実現により〜〜が可能となる | T2 / I4 |
| 7 | 本研究の実験では〜〜を構築し、有効性と実現可能性を検証した | T2末尾 / E1予告 |

- **1章への展開**：順番を少し入れ替え、各文を数文ずつに膨らませて具体例・説明を加える（骨組みは同じ）。
- **検査観点（review/audit共通）**：前半4文が成立しているか＝「人間の機能→未実現→重要課題→提案」の閉世界が読み取れるか。世界観が広がりすぎて前半戦が長文化していないか。
- **箱テンプレ骨格では適用しない**：`paper-composer.md` のアブスト①〜⑥定型が正（論理構造は7文法と同型なので矛盾しない）。閉世界の検査観点のみ共有する。

## T：Title / Abstract / Keywords

| ID | 何を書くか | ✅判定基準 | 充足質問の例 |

## I：Introduction

| ID | 何を書くか | ✅判定基準 | 充足質問の例 |
|---|---|---|---|
| I1 | 重要性の宣言 | 対象現象・活動がなぜ重要か。**7文法・1文目**（人間ならこうする／人間には難しい）から入り、技術詳細に入らない | 「この研究が解く問題は、誰にとって・なぜ今重要ですか？」 |
| I2 | 問題とギャップ | ①現象の定義→②容易に解決/分析できない現状（**7文法・2文目**：システム上で未実現）→③その根本理由→重要課題（**3文目**）、の論理チェーンが揃う | 「既存手法で③（できない理由）は何ですか？」 |
| I3 | アプローチ宣言 | "In this paper, we propose/present 〈手法名〉" の一文＋何を実現するか | 「手法の固有名詞（大文字始まり）は確定していますか？」 |
| I4 | 貢献リスト | 番号付きで各項目が独立に検証可能。Eブロックの実験と1対1対応 | 「貢献を2〜4項目に分けるとどうなりますか？各項目を裏付ける実験は？」 |
| I5 | 構成案内 | "Section 2 ... Section N ..." が実際の章番号と一致 | —（自動生成可能） |

## M：Methods（★型で差し替え）

| ID | 何を書くか | ✅判定基準 |

| MA1 | 上位概念の定義 | 概念が何を実現するかを1〜2文で宣言し、Overview図を参照 |
